## 4. Feasibility: does the shape survive into `scripts/stage2-retag/`?

**Yes.** The eval script already IS the call shape running standalone under Node with the same SDK version spec — the Stage-2 runner is "a sibling of `eval-llm-tagging-prompt.ts`" exactly as the exploration put it. Per-field mode is literally today's shape (one prompt + one vocab + one forced tool per call). Nothing in the canonical shape (client construction, model param, system+tool `cache_control`, enum-in-`input_schema`, forced `tool_choice`, single-user-turn body, `tool_use` extraction, post-hoc Zod) needs to change.

What a Stage-2 mirror must **ADD** (additions, not changes):

1. **Corpus iteration** — fetch lesson bodies (`lessons.content_text`, PROD) instead of local `--samples` JSON. Neither site touches the DB for input today.
2. **Per-field vs monolithic orchestration (OQ3)** — eval parameterizes exactly one field per run. Per-field Stage 2 = N sequential/parallel calls per lesson reusing today's shape verbatim. Monolithic = one tool whose `input_schema` has one enum-constrained property per field — schema construction grows, call shape doesn't change. Both shapes are expressible in the existing pattern; OQ3 is purely token-economics.
3. **Vocab adapter** — the new canonical vocab files (`data/vocab/cultural-heritage.vocab.json`, `data/vocab/academic-concepts.vocab.json`, emitted 2026-06-11/12 from the Stage-1 worksheets) have shape `{provenance, canonical[], alias_map, drops}` — NOT the eval harness's `{name, mode, values}` shape. A small adapter must derive the flat enum value list for `input_schema` injection (and the Session-1 throwaway harness needs the same adapter or hand-flattened vocab files).
4. **Batch API path (OQ4, if chosen)** — neither site uses `client.messages.batches.create`. The Batches API takes the identical per-request params under `{custom_id, params}`, so this is additive; carries the exploration §5 item 7 retention caveat (~29-day batch data retention, not ZDR-eligible) and best-effort caching.
5. **JSONL artifact persistence + resume** — eval does one whole-run `writeFileSync` at the end (`:450`); a corpus runner needs append-per-lesson JSONL (id, predicted, usage, model, prompt identity) and resume-by-id skip.
6. **Per-record cost accounting** — extend eval's `totalUsage` pattern to per-record; projected-full-run-cost math for the OQ3 dry-run.
7. **Long-run retry/checkpoint semantics** — SDK default 2 retries is fine per-call; run-level resumability is the real mechanism (item 5).
8. **Dry-run / diff / apply gates + rollback artifacts** — runner-level orchestration, out of the call shape entirely (and the reason the exploration says don't host this in an edge function).
9. **Check surface (OQ12)** — `tsconfig.scripts.json` + ESLint override + unit tests, since today's surface excludes scripts (see §5).

Nothing requires extending `process-submission`. The runtime-vs-call-shape split asserted in the exploration (§4 "Critical refinement") is confirmed by the code as it stands.
