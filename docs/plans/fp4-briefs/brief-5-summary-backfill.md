# FP4 Brief 5 — Summary backfill, extract-first (M–L)

Read `docs/plans/fp4-briefs/README.md` (standing rules) first. Evidence: **C4/FP4-DB-02**
plus the live-drive "---" note in `docs/plans/fp4-discovery/discovery-evidence.md`.

## Facts (Fable-probed on PROD, 2026-07-03 — re-probe before acting)

- **65 active** lessons (of 703) have a blank `summary` (empty string after btrim; 0 NULL).
  21 retired blanks are OUT of scope.
- `metadata` JSONB has **no `summary` key on any of the 785 rows** → the column is the only
  representation; do NOT touch `metadata` in this brief. (The column-vs-JSONB mirror gotcha
  does not apply to summary — verified.)
- All 65 have `content_text` > 200 chars — extraction has material to work with.
- UI reads `lesson.summary` (IntLessonDetail.tsx:96 + search cards).
- One active lesson is literally titled `---` (sorts first under Title A–Z); several titles
  carry trailing control characters (vertical tab) — census these too.

## Owner decision (2026-07-03, verbatim intent)

Summaries should be **taken from the lesson text whenever possible** — most lesson plans
already contain one. LLM-drafting is the fallback only where nothing extractable exists.
Fix the `---` title in the same pass. The owner reviews everything before any write.

## Phase 1 — produce the review doc, then STOP

1. Census the 65 rows (id, title, `content_text`). For each, look for an existing
   summary/description/overview section or a usable opening paragraph. Classify:
   - **EXTRACTED** — verbatim (lightly trimmed) from the lesson's own text. Prefer this.
   - **DRAFTED** — 1–2 plain sentences you write from the content, only when nothing
     extractable exists.
   Keep summaries in the register of existing ones (read ~10 populated `summary` values
   first and match tone/length). Curriculum-facing copy stays plain — no jargon, no
   marketing voice.
2. Title hygiene census: the `---` row (propose a real title from its content) + every
   title with leading/trailing whitespace or control chars (propose the trimmed form).
3. Write `docs/plans/fp4-briefs/brief-5-summaries-review.md`: one table —
   lesson title | proposed summary | source (EXTRACTED/DRAFTED) — plus the title-fix list.
   Include the census SQL + raw counts + which DB (provenance is committed).
4. **STOP. Hand back the file path.** The owner (with Fable) reviews and edits that doc.
   No writes of any kind in Phase 1.

## Phase 2 — after explicit owner approval of the reviewed list

1. Build the data fix as one committed SQL file (`brief-5-summary-data-fix.sql` in this
   dir): single statement, `UPDATE lessons SET summary = v.summary FROM (VALUES …) v(id,
   summary) WHERE lessons.lesson_id = v.id`, plus the approved title fixes. Only rows from
   the approved doc, ids verbatim.
2. **NOT-VALID CHECK gotcha:** any UPDATE re-validates `valid_cooking_skills` and
   `valid_main_ingredients` against the whole row. Active-row violators were 0 on
   2026-07-03 (B5 healed them) — re-probe that immediately before applying; if any of the
   target rows would trip either check, STOP.
3. Rehearse on TEST with a rollback transaction; post-asserts: 0 active blank summaries
   among approved rows, titles clean, row counts untouched, spot-check 3 rows byte-equal to
   the approved doc. Then restore and confirm baseline.
4. PROD apply is **owner-gated, owner presses/approves in-channel**; immediately before it,
   re-run the pre-probes (blank count, violator count). Post-apply: same asserts on PROD.
5. Note in the PR/hand-back (do not fix): `content_embedding` is NOT regenerated by this
   change (embeddings-regen is a deferred campaign item, C2.4), and report whether the FTS
   vector includes `summary` (check the tsvector/trigger definition and say which fields it
   covers — if summary is indexed, these lessons become findable by their new summaries,
   which is desired, just say so).

## Verify

Phase 1 has no code gates (docs only). Phase 2: TEST rehearsal evidence + per-round TEST
re-verification if review iterates; `npm run check`/`test:run` only if any code changed
(none expected).

## STOP conditions

Blank-summary count ≠ 65 at census time (re-census, report delta); any lesson whose content
has no extractable material AND resists a faithful 1–2 sentence draft (flag it in the review
doc as SKIPPED-needs-owner instead of forcing copy); anything requiring a schema change.
"STOP = write the hand-back and END YOUR TURN; design forks route to Fable; the owner only
answers explicit approvals (data fix / merge / gates)."
